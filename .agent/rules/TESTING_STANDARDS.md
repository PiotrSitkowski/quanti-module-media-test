## Core Principles

### 1. Tests First (TDD)
Write tests BEFORE the implementation. Derive test cases from:
- `MODULE_BRIEF.md` — business requirements and domain rules
- `definition.ts` — slots, permissions, behavior rules, data semantics
- `contract.ts` — Zod payload schemas (test validation edges)

### 2. No Excessive Mocking
Mock ONLY external dependencies that are unavailable in the test environment:
- Cloudflare D1/R2 bindings → mock the `env.BACKEND.sys_executeDbQuery()` proxy
- External HTTP calls → mock `fetch()`
- Queue producers → mock `env.QUEUE_MAIN.send()`

**NEVER mock:**
- Zod schema validators — they must run against real code
- Business logic mapping functions — test with real data
- React component rendering — use `@testing-library/react`

### 3. Black-Box Testing
Test through the public API surface, not internal implementation details:
- **Worker methods:** call `list()`, `create()`, etc. with full payloads
- **React components:** render with props, assert visible output
- **Zod schemas:** call `.safeParse()`, assert success/failure

### 4. Sad Path Ratio: 2:1 Minimum
For every test of a correct input (Happy Path), write at least TWO tests
for error cases (Sad Paths):
- Missing required fields (e.g., no `projectId` → tenant isolation breach)
- Invalid data types (e.g., string where number expected)
- Boundary values (e.g., empty string, zero, negative numbers)
- Unauthorized access patterns (e.g., wrong `instanceKey`)

### 5. File Naming Convention
```
src/worker.test.ts                    — worker unit tests
src/components/MyComponent.test.tsx   — component tests (MFE)
contract.test.ts                      — contract/schema edge-case tests
```

### 6. Deploy Gate
`quanti deploy` runs `npm run test` before any upload.
**If tests fail → deploy is BLOCKED. No exceptions.**
